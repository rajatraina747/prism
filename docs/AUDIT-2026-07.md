# Security & Performance Audit — July 2026 (v1.6.0)

Point-in-time review of the full codebase (Rust backend, React frontend,
extension, build config) plus `cargo audit` / `npm audit`. Work these down
over time; check items off here and mirror headline items in ROADMAP.md.

## Security

### What's already solid (verified, no action)

- **Argument injection**: every yt-dlp invocation appends the URL after a `--`
  options terminator (`lib.rs` parse_url/parse_playlist, `download_manager.rs`).
- **Path traversal**: `validate_download_path` rejects `..` components,
  requires absolute paths, canonicalizes the deepest existing ancestor
  (symlink-safe), confines to home/Downloads/appdata. `validate_open_path`
  re-validates anything the webview asks to open/reveal.
- **Template injection**: `sanitizeFilename` escapes `%` (yt-dlp template
  syntax), strips `/ \` and control chars.
- **settings.json as trust boundary**: every Rust-side read is whitelisted
  (browser names, proxy schemes, audio formats, tracker/blocklist schemes).
- **Supply chain**: yt-dlp self-update = SHA-256 vs release manifest +
  test-run + atomic rename; app updates minisign-signed; Sentry doubly opt-in.
- **Webview**: CSP set; no dangerouslySetInnerHTML/eval; deep links validated
  http(s) on both extension and app sides; fs capability scoped to appdata.

## Performance

| # | Impact | Finding | Fix |
|---|--------|---------|-----|
| P1 | High | yt-dlp fragment concurrency (`-N`) unused — HLS/DASH downloads run one connection and lose 2–4× throughput vs 4KVD/IDM-class tools. | Add `-N 4` in `download_manager.rs` (consider a setting). |
| P2 | Medium | Progress emits per stdout line → IPC + reducer `queue.map` + provider re-render several times/sec per active download. Fine at 3 concurrent, janky at 10+. | Throttle emits Rust-side to ~4/s per item. |
| P3 | Medium | Auto-start effect in `AppProvider.tsx` depends on `queue`, so it re-runs (filter + scheduleGate) on every progress tick. | Key on a derived value (statuses/ids) instead of the whole queue. |
| P4 | Medium (future) | Library renders up to 2,000 history rows, no virtualization/pagination. | Virtualize or paginate when history fills. |
| P5 | Low | `history.json` rewritten in full, undebounced, on every completion (queue writes are debounced 300ms). | Debounce like the queue. |
| P6 | Low | `@sentry/react` statically imported — bundle cost even when reporting is off. | Dynamic-import inside `syncCrashReporting`. |
| P7 | Low | Forced `--remux-video mp4` can put VP9/AV1 into .mp4, which some players mishandle. Also a compat/UX issue. | "Keep original container" option (mkv/webm) alongside MP4-compatible default. |

## Postmortem — the 4K-that-downloaded-1080p bug (2026-07-16, fixed in 1.7.0)

Selecting 2160p silently delivered 1080p: the per-resolution yt-dlp chain put
`bestvideo[height<=H][vcodec^=avc1]` first, and yt-dlp takes the FIRST
satisfiable alternative — avc1 satisfies `<=2160` at 1080p (YouTube's H.264
stops there), so the any-codec alternative holding real 4K was never reached.

**Why two audits missed it:** the security pass read those format strings for
injection, not behavior; the performance pass looked at throughput, not
output correctness. Nothing fails — no error, no crash, green tests — and
catching it statically required simulating yt-dlp's selector semantics against
YouTube's codec/resolution matrix. Classic *silent degradation*: every
observable signal reports success.

**Lesson:** user-facing claims deserve verification, not just code paths.
Guards added in 1.7.0: (a) exact-height-first format chains; (b) yt-dlp now
prints the delivered height (`--print after_move:` + `--no-quiet`), completion
carries `actual_height`, the UI warns on shortfall and the Library shows
"1080p (asked 2160p)" in amber. The mismatch surface converts this whole bug
class — any site, any future regression — into a visible signal.
